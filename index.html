⚠ Worst-Case Scenario

For Dijkstra with a binary min-heap, the worst case is a complete graph where:

  • Every vertex connects to every other: E = V(V-1)
  • Every edge triggers a DECREASE-KEY: E log V heap operations
  • All V vertices are extracted from the heap: V log V extractions

Total: O((V + E) log V) = O(V² log V) for a complete graph.

For such dense graphs the O(V²) array implementation is actually faster, since log V > 1 and E ≈ V².

⛔ Why Dijkstra Fails with Negative Weights

Dijkstra's greedy guarantee is: "Once a node is extracted from the priority queue, its distance is final."

This holds only if all edge weights are ≥ 0. Here's why it breaks:

Graph:  A →4 B →3 D
        A →2 C →-5 B

Dijkstra's steps:

  1. Extract A (dist=0). Relax: dist[B]=4, dist[C]=2.
  2. Extract C (dist=2). Relax: dist[B] = 2+(-5) = -3. But B may already be finalized!
  3. If B was extracted at dist=4 before C was processed, we miss the true shortest path dist[B]=-3.

Fix: Use Bellman-Ford, which relaxes all edges V-1 times, catching negative-weight updates regardless of order. It runs in O(VE) instead of O((V+E) log V).

⚠ What Happens Without a Priority Queue?

Scenario 1: Replace PQ with a simple unsorted array
Instead of EXTRACT-MIN in O(log V), we scan all unvisited nodes linearly: O(V) per step.
Total: O(V²) instead of O((V+E) log V).
For a graph with 1000 nodes and 3000 edges: heap = ~33,000 ops vs array = ~1,000,000 ops. 30× slower.
Scenario 2: Remove the PQ entirely (process nodes in arbitrary order)
Without always picking the closest unvisited node, the greedy-choice property breaks.
A node might be processed with a non-optimal distance, then its neighbors get wrong values propagated.
The result: incorrect shortest paths — the algorithm is no longer Dijkstra at all.
Scenario 3: Use a max-heap instead of min-heap
EXTRACT-MAX always picks the farthest unvisited node first.
This explores the worst paths first, leading to incorrect distance updates. Shorter paths through closer nodes are never explored early enough to improve longer ones.
This is equivalent to running BFS backwards in terms of priority.
Summary: Why EXTRACT-MIN + DECREASE-KEY are critical
EXTRACT-MIN enforces the greedy choice — process closest node first.
DECREASE-KEY ensures the PQ reflects updated (shorter) distances found during relaxation.
Together they guarantee: when a node is extracted, its distance is provably optimal.
✓ Greedy Algorithm
Dijkstra is greedy: at every step it commits to the globally unvisited vertex with the smallest known distance, without reconsidering past choices. This works because non-negative edge weights guarantee that no future path can improve an already-finalized node.

Proof: Why the Greedy Choice is Always Correct

Theorem (Dijkstra's Correctness): When a vertex u is extracted from the priority queue, dist[u] is the true shortest-path distance from the source.


Proof by contradiction:

1Suppose u is extracted with dist[u] = d, but the true shortest distance is d* < d.
2There must exist an alternative path s → … → x → y → … → u with total weight d*.
3Let x be the last visited node on this path, and y the first unvisited node. Since dist[x] is final, dist[y] ≤ dist[x] + w(x,y) ≤ d* < d.
4But then dist[y] < dist[u], so the priority queue would have extracted y before u — contradiction! u cannot have been extracted first.
5Therefore dist[u] = d* — the greedy choice is always optimal. This relies on all edge weights ≥ 0 (step 3 requires w(x,y) ≥ 0).

⛔ Counterexample: When Greedy Fails

On a graph with a negative-weight edge, the greedy assumption breaks. Dijkstra finalizes a node too early — before discovering a cheaper path through the negative edge.

Graph: A →4 B,  A →2 C,  C →−5 B


Dijkstra (wrong):

  1. Extract A (d=0). Set dist[B]=4, dist[C]=2.
  2. Extract C (d=2). Relax: dist[B] = 2+(−5) = −3. Update dist[B].
  3. Extract B (d=−3). ← Dijkstra may report dist[B]=4 if B was finalized in step 1 before C relaxed it!

The issue: if dist[B]=4 was already finalized (extracted) before C was processed, the update to −3 comes too late.


Bellman-Ford (correct): Runs V−1 full relaxation passes. In pass 2 it re-relaxes B via C, correctly finding dist[B] = −3. Dynamic Programming does not commit early — it re-examines everything.

🗺 Overview — What is Dijkstra's Algorithm?
▼

Dijkstra's algorithm, published by Dutch computer scientist Edsger W. Dijkstra in 1956, solves the Single-Source Shortest Path (SSSP) problem: given a weighted graph and a source vertex, find the shortest path from the source to every other vertex.

What problem does it solve?

  • Navigation apps (Google Maps, GPS routing)
  • Network routing protocols (OSPF uses Dijkstra)
  • Game AI pathfinding (finding optimal routes on a map)
  • Social network analysis (shortest connection between people)
  • Airline route optimization

How does it work? (Plain English)

  1. Start at the source node. Set its distance to 0. Set all others to infinity (unknown).
  2. Put all nodes in a priority queue ordered by their current distance.
  3. Repeatedly pick the closest unvisited node (greedy choice).
  4. For each of its neighbors, check: "Is going through this node shorter than what we know?" If yes, update the neighbor's distance (relaxation).
  5. Mark the node as visited — its distance is now final and optimal.
  6. Repeat until all nodes are visited or the queue is empty.

When should you use it?

  • Use Dijkstra when: all edge weights are non-negative, you need single-source shortest paths, and speed matters.
  • Use Bellman-Ford when: the graph may contain negative-weight edges.
  • Use Floyd-Warshall when: you need all-pairs shortest paths.
  • Use A* when: you have a single target and a good distance heuristic (e.g., Euclidean distance).

Key Requirement

All edge weights must be ≥ 0. With negative weights, the greedy property breaks — a seemingly finalized node can later be improved via a negative edge, making previously calculated distances wrong.

💡 Key Concepts
▼
Edge Relaxation
For edge u→v with weight w: if dist[u]+w < dist[v], update dist[v]. This is called relaxing the edge. Every shortest path is built by a sequence of successful relaxations.
Greedy Choice Property
The node extracted from the PQ always has its true shortest distance. No future path can improve it — because all remaining edges have non-negative weight, so future paths can only be equal or longer.
Optimal Substructure
Every sub-path of a shortest path is itself a shortest path. If A→B→C is the shortest path from A to C, then A→B must be the shortest path from A to B.
Priority Queue Role
The min-heap always surfaces the globally cheapest unvisited node. This is what makes Dijkstra efficient — O(log V) per extraction instead of O(V) for a linear scan.
Negative Weights Break It
Once a node is finalized (extracted), Dijkstra never revisits it. A negative-weight edge discovered later could have produced a shorter path — but Dijkstra already committed. Bellman-Ford avoids this by relaxing all edges V−1 times.
Path Reconstruction
The prev[] array records the predecessor of each node on the shortest path. To get the path to any node v: start at v, follow prev[v], prev[prev[v]], etc. back to the source.
⚠️ Common Mistakes & Misconceptions
▼
❌
Mistake: Using Dijkstra on a graph with negative edges.
Dijkstra finalizes distances as it goes. A negative edge encountered later can invalidate an already-finalized distance, but Dijkstra won't go back to fix it.
Fix: Use Bellman-Ford for graphs with negative weights (but no negative cycles).
❌
Mistake: Thinking the shortest path uses the fewest edges.
Dijkstra minimizes total weight, not hop count. A 3-hop path with total weight 5 beats a 1-hop path with weight 10.
Fix: For minimum hops, use BFS (unweighted) or set all weights = 1.
❌
Mistake: Visiting a node more than once.
Once a node is extracted from the PQ (finalized), its distance is optimal. Re-processing it wastes time and produces incorrect results in some implementations.
Fix: Always check if visited[u]: continue before processing a node.
❌
Mistake: Confusing Dijkstra with BFS.
BFS explores nodes level by level and works for unweighted shortest paths. Dijkstra uses a priority queue to always explore the cheapest node next — a different strategy.
Fix: BFS = shortest path by hops; Dijkstra = shortest path by weight.
❌
Mistake: Forgetting to initialize dist[source] = 0.
If source starts at ∞ like all other nodes, it will never be extracted first and the algorithm produces wrong results.
Fix: Always set dist[source] = 0 before building the priority queue.
❌
Mistake: Using a max-heap instead of a min-heap.
A max-heap extracts the farthest node first — the opposite of what Dijkstra needs. This produces completely wrong shortest paths.
Fix: Always use a min-heap (minimum distance at the root / top).
🔀 Algorithm Variants & Extensions
▼

Dijkstra with different data structures

  • Simple array: O(V²) time — best for dense graphs (E ≈ V²)
  • Binary min-heap: O((V+E) log V) — best for sparse graphs
  • Fibonacci heap: O(V log V + E) — theoretically optimal, rarely used in practice due to high constants

Bidirectional Dijkstra

Run Dijkstra simultaneously from source AND target. Stop when the two frontiers meet. Roughly 2× faster than standard Dijkstra for single-pair queries. Used in real GPS systems.

A* Search

Dijkstra extended with a heuristic function h(v) that estimates the remaining distance to the target. Uses priority = dist[v] + h(v) instead of just dist[v]. Explores fewer nodes when the heuristic is good (e.g., straight-line distance for geographic graphs).

Johnson's Algorithm

Handles negative weights by re-weighting edges to be non-negative (using Bellman-Ford once), then running Dijkstra from every vertex. Total: O(V² log V + VE) — better than V × Bellman-Ford for sparse graphs.

SSSP on DAGs

For Directed Acyclic Graphs, topological sort + single relaxation pass gives O(V+E) — faster than Dijkstra and works with negative weights.
